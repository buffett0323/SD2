Motivation

The parser wants a prefix. Diffusion gives holes.

0:20-0:50Problem

Diffusion denoising state

{ "method" : MASK , "expIds" : MASK }

The model predicts multiple masked positions while seeing both left and right context.

Grammar checker expectation

{ "method" : "get" , "expIds" : ["1a...ef"] }

A parser can advance cleanly only when the left-to-right prefix is concrete.

Why it matters

  • JSON APIs, tool calls, code, and structured records reject malformed output.
  • Autoregressive constrained decoding assumes a stable left prefix.
  • Diffusion LLMs are attractive because they refine many tokens in parallel.
  • The open question is how to impose grammar constraints without throwing away that parallelism.
Core tension: bidirectional model proposals versus left-to-right grammar state.

Prior Works

Existing methods solve pieces, but hard schemas expose a gap.

0:50-1:15Context

AR constrained decoding

Outlines, XGrammar, FSM-based decoding. Strong when tokens arrive left-to-right.

->

Diffusion constrained decoding

CD4dLLM and DINGO bring grammar ideas to masked diffusion models, often with higher checking cost.

->

LAVE

Lookahead-then-verify samples random suffixes. It works well on easier JSON tasks, but hard schemas make valid suffixes rare.

Missing piece: deterministic repair for multi-token joint constraints, such as an enum choice and a regex-constrained value that must become valid together.
Prior work motivates the baseline: LAVE is strong, but stochastic validation can falsely reject valid paths.

Approach Overview

Two steps: deterministic frontier stack, then joint span repair.

1:45-2:25Methods
LAVE baseline
Random lookahead verification.

Sample K suffixes at the frontier. Accept a token only if at least one completion is valid.

Dgrammar
Deterministic frontier masking and incremental parser state.

Use llguidance to mask only the first unresolved token, commit adaptive batches, remask exact violators, and overlap CPU mask computation with GPU forward.

DPGrammar
Viterbi repair over the violated span.

When greedy repair fails, solve a small grammar-state DP across the span instead of fixing one token at a time.

Engineering choices

A

Frontier mask: compute valid next tokens from the current LLMatcher state.

B

AIMD batching: grow commit size after success; reset after grammar conflict.

C

Selective remask: remask only the first offending token, not the whole prefix.

D

Async overlap: build the next mask on CPU while GPU forward runs.

Dgrammar raises validity and removes timeouts, but greedy single-token repair still misses correlated errors.

Method 2

DPGrammar treats a violation span as one joint decision.

3:05-3:50Method

Greedy repair

t_c ... t_s

Fixing t_c may leave t_s invalid. The decoder can bounce through repeated resampling.

t'_c ... t_s

Viterbi span repair

Step 0q0q1q2q3q4
Step 1q0q1q2q3q4
Step kq0q1q2q3q*

DP state is a grammar automaton state. Paths that reach the same state are merged, keeping the highest log-probability path.

Search objective: maximize token log probability subject to grammar feasibility across the whole span.

Interpretation

  • LAVE -> Dgrammar: removing random lookahead eliminates timeouts and improves validity.
  • Dgrammar -> DPGrammar: joint span repair handles multi-token constraints that greedy remasking misses.
  • Cost tradeoff: DP work is added only on violation spans, not across the whole sequence.

Discussion

What worked, what did not, and where the method is sensitive.

5:10-5:40Discussion

What worked

  • Incremental parser state avoids repeated full-prefix checks.
  • Frontier masking keeps deterministic grammar guarantees local.
  • AIMD batching preserves some diffusion parallelism.
  • Viterbi repair finds consistent multi-token replacements.

Limitations

  • 75 schemas are skipped because llguidance cannot compile them.
  • DPGrammar still misses 4 out of 511 matched instances.
  • Top-K DP can miss a valid path outside the candidate set.
  • Functional correctness beyond schema validity is not the focus here.
Design choice: solve the short violated span rather than running full-sequence DP every step.

Conclusion

The engineering layer gives the frontier; the DP layer gives the correction.
